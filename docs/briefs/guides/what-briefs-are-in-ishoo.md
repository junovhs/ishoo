# Purpose
Briefs capture the durable project understanding that Ishoo should not force the team to keep rediscovering.

They are the structured layer for decisions, principles, research syntheses, questions, insights, and guides that shape future work.

# Scope
Briefs are for:
- decisions that affect future implementation
- principles that should guide repeated choices
- research that has been distilled into project-relevant conclusions
- insights that are likely to matter again
- questions important enough to track explicitly
- guides about app boundaries, philosophy, and operating rules

# Non-goals
Briefs are not:
- raw notes
- AI transcript dumps
- generic longform docs
- issue replacements
- a separate standalone app
- a place to store everything

# Relationship to Issues
Issues track actionable work.

Briefs track the reasoning, conclusions, and current understanding that inform work.

A Brief may spawn Issues.
An Issue may create or refine Briefs.
They are adjacent but not the same artifact.

# Relationship to the Suite
Ishoo owns work and project memory.
SEMMAP owns structural understanding of the codebase.
Neti owns enforcement and constraint checking.

Briefs live inside Ishoo because the user should not need to adopt a fourth app to benefit from project memory.

# Operating Rule
If an artifact implies work, it should be an Issue.
If an artifact improves judgment, preserves rationale, or clarifies project understanding, it should be a Brief.

# Update Policy
This guide should change rarely.
If the concept of Briefs shifts materially, update this Brief and create a linked Decision Brief explaining why.
