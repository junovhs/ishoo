## Compositor-Driven Scroll in Embedded Webviews: Architecture and Techniques

### 1. The Core Problem: Webview Threading Model

The fundamental challenge in embedded webviews stems from how the compositor thread operates. A key use of the compositor thread is to scroll pages smoothly even when the main thread is blocked. This is done by intercepting input events before they arrive on the main thread's event loop and redirecting them onto the impl thread.

However, embedded webviews introduce complications. The Chrome compositor has a software mode, but that on its own doesn't solve all the uses/abuses of existing Android WebView embedders. More critically, single-threaded versions of cc do not have a pending tree and commit directly to the active tree. This is an optimization to avoid extra work and copies. To work around this, the active tree is unable to be drawn until its tiles are all ready to draw. However, given that this is a single-threaded version of cc, there are no compositor thread animations or scrolling, and so there is little reason to need to draw.

**This is the critical insight**: if your embedded webview is running cc in single-threaded mode (which some webview embeddings do), you get *zero* compositor-thread scrolling. You must verify your webview is running in threaded compositing mode.

### 2. What Forces Scroll Back to the Main Thread

There are specific reasons the compositor falls back to "slow" main-thread scrolling:

**a) Non-Fast Scrollable Regions (event handlers):**
Since running JavaScript is the main thread's job, when a page is composited, the compositor thread marks a region of the page that has event handlers attached as "Non-Fast Scrollable Region". If you look at this code from the browser's point of view, now the entire page is marked as a non-fast scrollable region. This means even if your application doesn't care about input from certain parts of the page, the compositor thread has to communicate with the main thread and wait for it every time an input event comes in. Thus, the smooth scrolling ability of the compositor is defeated.

**Mitigation**: You can pass `passive: true` options in your event listener. This hints to the browser that you still want to listen to the event in the main thread, but compositor can go ahead and composite new frame as well.

**b) Uncomposited scrollers:**
The design works only on composited scrollers. If the scroller is uncomposited, we don't get a cc::Layer for it nor its scrollbars so there's nothing for us to grab there. The goal is to allow composited scrollers to scroll on the compositor thread.

**c) Synchronous JS handlers:**
Some input can't be handled by the compositor thread (e.g. there's a synchronous Javascript touch or wheel handler) and so that input is forwarded along to Blink to handle directly.

### 3. `position: sticky` and the Compositor

Sticky positioning historically had a troubled relationship with the compositor. Chrome once removed `position:sticky` because "the current implementation isn't designed in a way that integrates well with the existing scrolling and compositing system". They stated: "Once we've got our scrolling and compositing house in order, we should return to position: sticky and implement the feature in a way that integrates well with the rest of the engine".

In modern Chromium, sticky elements interact with the compositor via **property trees**. Chrome's Compositor-Thread Scrolling architecture achieves smooth, low-latency scrolling by handling most scroll events off the main thread. This process relies on key data structures called property trees—specifically the ScrollTree, TransformTree, and ClipTree. The LayerTreeHostImpl orchestrates the sequence: it receives input events via the InputHandlerProxy, determines the scroll target via hit testing, updates the ScrollNode's offset, and leverages the TransformTree to translate content on the GPU without needing the main thread for layout or style updates.

Compositing category (3) triggers based on whether composited scrolling applies to the LayoutObject, or the LayoutObject moves relative to a composited scroller (`position: fixed` or `position: sticky`).

**Best practice for sticky elements:**
Scrolling elements containing fixed or sticky content can cause performance and accessibility issues. As a user scrolls, the browser must repaint the sticky or fixed content in a new location. Depending on the content needing to be repainted, the browser performance, and the device's processing speed, the browser may not be able to manage repaints at 60 fps. One solution is to add `will-change: transform` to the positioned elements to render the element in its own layer, improving repaint speed and therefore improving performance and accessibility.

### 4. `will-change` and Layer Promotion

`will-change: transform` is the primary mechanism for explicit compositor layer promotion:

The `will-change: transform` property is key. `will-change` is an official web standard that instructs a browser to render the element independently of its surroundings. Used sparingly, this property allows us to say "hey browser, this element will change somehow in the future, please paint it on its own unique layer so that its surroundings don't affect how it is painted."

Layers can be visually updated independently by a separate thread, the Compositor Thread, without competition from heavy JavaScript or other activity taking place on the Main Thread. This is a tradeoff, where GPU memory is traded to optimize rendering performance.

**Critical caveats:**

- It's tempting to promote all elements with `* { will-change: transform; }`, but every layer you create requires memory and management, and that's not free. On devices with limited memory the impact on performance can far outweigh any benefit of creating a layer.

- Do not add the `will-change` property to too many elements. The browser tries to perform optimization on its own and excessive use can lead to high resource consumption. Adding the property in CSS causes the browser to keep the optimization in memory long beyond what is really necessary. It is preferable to use JavaScript to add the property just before the element changes and remove it after completion.

For animations specifically, the best-performing pixel pipeline avoids both layout and paint, and only requires compositing changes. Today there are only two properties for which that is true—transforms and opacity. The caveat is that the element on which you change these properties should be on its own compositor layer.

### 5. `contain` and CSS Containment

The key and overarching goal of CSS containment is to enable rendering performance improvements of web content by providing predictable isolation of a DOM subtree from the rest of the page. A developer can tell a browser what parts of the page are encapsulated as a set of content, allowing the browsers to reason about the content without needing to consider state outside of the subtree. Knowing which bits of content (subtrees) contain isolated content means the browser can make optimization decisions for page rendering.

There are four types of CSS containment: `size` containment ensures the element's box can be laid out without examining its descendants. This means we can potentially skip layout of the descendants if all we need is the size of the element.

How `contain` interacts with the compositor thread delegation:

- **`contain: layout paint`** — isolates the subtree's layout and paint from siblings. This doesn't directly promote to a compositor layer, but it *constrains the blast radius* of main-thread work. When the compositor thread hands scroll deltas back to the main thread during a BeginMainFrame, BeginFrameArgs contain scroll deltas applied on the compositor thread. Blink applies those deltas, kicks off requestAnimationFrame logic, and finishes the Blink half of the rendering lifecycle. With `contain: layout paint`, Blink can scope this work to the contained subtree.

- **`contain: strict` (= `size layout paint style`)** — provides the strongest isolation. Combined with an explicit height, this allows the engine to entirely skip layout for offscreen contained regions, reducing the probability that the main thread stalls long enough to starve the compositor.

- `contain` does **not** guarantee layer promotion — it reduces main-thread work so that the compositor can more reliably "win the race" during threaded scrolling.

### 6. `content-visibility` and Rendering Skipping

In order to realize the benefits of `content-visibility`, the browser needs to apply size containment to ensure that rendering results of contents don't affect the size of the element in any way. This means the element will lay out as if it was empty. If the element does not have a height specified in a regular block layout, then it will be of 0 height.

`content-visibility: auto` interacts with compositor delegation **indirectly**:

- It reduces the amount of work in the **paint** and **layout** stages for offscreen content, meaning `BeginMainFrame` completes faster.
- Animations of visual effects, and scroll, can skip layout, pre-paint and paint. If layout, pre-paint, and paint can be skipped for visual effects, they can be run entirely on the compositor thread and skip the main thread.
- By making offscreen regions effectively invisible to the rendering pipeline, `content-visibility: auto` reduces the total commit cost. If cc determines that no work is required (e.g. a compositor thread scroll updates Blink, but Blink makes no changes to the page in response to that scroll), then it may abort the commit early. Lighter commits mean the compositor thread spends less time blocked waiting for the main thread.

**However**: `content-visibility: auto` requires `contain-intrinsic-size` to avoid scroll-bar jank. The size of the scrollbar will shift, being reliant on each story having a non-zero height. CSS provides `contain-intrinsic-size`, which effectively specifies the natural size of the element if the element is affected by size containment. This means it will lay out as if it had a single child of "intrinsic-size" dimensions, ensuring that your unsized divs still occupy space.

### 7. Practical Layering Strategy for Your Webview

Here is the recommended layering approach for a DOM with `position: sticky` headers and absolutely-positioned layers with CSS transitions:

```css
/* 1. Sticky header — promote to its own layer */
.sticky-header {
  position: sticky;
  top: 0;
  will-change: transform;       /* explicit layer promotion */
  contain: layout style;         /* isolate layout from descendants */
}

/* 2. Absolutely-positioned animated layers */
.abs-panel {
  position: absolute;
  will-change: transform, opacity; /* only the properties you animate */
  contain: layout paint;            /* scope paint invalidation */
}

/* 3. Offscreen heavy content sections */
.content-section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px; /* estimated height */
}

/* 4. Main scroll container — ensure composited scrolling */
.scroll-container {
  overflow-y: auto;
  /* Browser will usually auto-composite, but if needed: */
  will-change: scroll-position;
}
```

**Input handler discipline** — ensure all scroll/touch listeners are passive:
```js
container.addEventListener('scroll', handler, { passive: true });
container.addEventListener('touchstart', handler, { passive: true });
container.addEventListener('wheel', handler, { passive: true });
```

### 8. Webview-Specific Concerns

**Chromium-based webviews (CEF, Electron, Android WebView):**
- In Chromium, there is a single global UI thread in the browser process. It is one of the explicitly named BrowserThreads. The UI thread is generally where UI code runs, including input delivery from the OS. In Android WebView specifically, input routing may funnel through the Android view system before reaching the compositor.
- Verify threaded compositing is enabled. Android WebView historically used synchronous compositing; the compositor can operate in either single or threaded mode. In single threaded mode, there is still a two-tree architecture and delayed commits, but a different synchronization/scheduling algorithm hosts the tree on the main thread. Single-threaded mode means no off-main-thread scroll.
- The Compositor thread processes input events, performs scrolling and animations of web content, computes optimal layerization of web content, and coordinates image decodes, paint worklets and raster tasks. All of this requires the threaded mode.

**WebKit-based webviews (WKWebView on macOS/iOS):**
- The Threaded Compositor was built to improve performance of WebKitGTK+ for CSS Animation using a dedicated thread for compositing. But not only CSS Animation, it also provides a way to improve the performance of scrolling, scaling, rendering of canvas and video element.
- WKWebView on iOS uses a different scrolling model where `UIScrollView` is the scroll driver and the web content is rendered into tiled layers. Layer promotion from `will-change` still applies, but the scroll coordination path is fundamentally different from Chromium's `InputHandlerProxy → LayerTreeHostImpl` pipeline. `content-visibility` support in WebKit is limited (check feature flags).

## 1. The Problem: Discrete Deltas from Hardware

Typical mouse wheel input is highly discrete, because at the level of hardware, mouse uses optical or mechanical switches to detect rotation. Standard detented wheels produce 120-unit increments (the Windows `WHEEL_DELTA` convention). Both spatial- and temporal resolutions of typical mouse wheel input is insufficient for smooth scrolling if we use that input to set scrolling position directly. However, that's exactly what most applications still do, and that's why scrolling in those applications is not smooth at all.

Even high-resolution "free-spinning" wheels don't fully solve this: Despite that high-precision scrolling is often touted as a means of smooth scrolling, it's only true for applications that use a naive approach to scroll event processing, and even then the "smoothness" is suboptimal and performance problems can be expected. As we have seen with the mouse pointer, both spatial- and temporal resolutions of input data are not enough to control the rendering directly and interpolation is the way to go.

## 2. Input Coalescing Strategies

### Browser-Level Coalescing (Pointer Events)

Modern browsers already coalesce continuous input events to align with the rendering frame. In Chrome 60, we are making a change that causes these events to occur at a lower frequency while increasing the granularity of the information provided. Much like when Jelly Bean was released and brought the Choreographer which aligns input on Android, we are bringing frame aligned input to the web on all platforms.

For pointer events, `getCoalescedEvents()` allows recovery of the full sub-frame history: The `getCoalescedEvents()` method of the PointerEvent interface returns a sequence of PointerEvent instances that were coalesced (merged) into a single `pointermove` or `pointerrawupdate` event. Instead of a stream of many `pointermove` events, user agents coalesce multiple updates into a single event. This helps with performance as the user agent has less event handling to perform, but there is a reduction in the granularity and accuracy when tracking, especially with fast and large movements. The `getCoalescedEvents()` method lets applications access all un-coalesced position changes for precise handling of pointer movement data where necessary.

### Application-Level Coalescing for Wheel Events

Wheel events don't have `getCoalescedEvents()`, so you must coalesce manually. The canonical pattern:

```js
let pendingDelta = 0;
let rafId = 0;

element.addEventListener('wheel', (e) => {
  e.preventDefault();
  pendingDelta += e.deltaY;         // accumulate
  if (!rafId) {
    rafId = requestAnimationFrame(flushScroll);
  }
}, { passive: false });

function flushScroll(timestamp) {
  rafId = 0;
  applySmoothedScroll(pendingDelta, timestamp);
  pendingDelta = 0;
}
```

This is critical because: Any rAFs queued in your event handlers will be executed in the same frame. Any rAFs queued in a rAF will be executed in the next frame. So multiple wheel events within one frame all add to `pendingDelta`, but only one rAF callback fires. Without this, it's up to you to coalesce rAFs on your own. So: if there's a chance of multiple of the "same" callbacks firing within the same frame, then you must manage the scheduling/coalescing.

**Important caveat on `passive`**: If the event is canceled, no scrolling or zooming is performed. This may cause performance issues as the browser has to wait for every wheel event to be processed before actually scrolling the content. You can avoid this by setting `passive: true` when calling `addEventListener()`, which may cause the browser to generate non-cancelable wheel events. Chrome even made wheel events passive by default on root elements — they are now passive by default for root elements in Chrome 73 (currently beta) which means React apps that have custom scrolling/zooming behaviors will run into issues. If you're taking over scrolling, you **must** use `{ passive: false }` on a non-root element to avoid fighting the compositor thread.

## 3. Smoothing Physics: Beyond Lerp

There are three major tiers, each with different latency/smoothness tradeoffs:

### Tier 1: Exponential Decay (Frame-Rate-Independent Damper)

Repeated calls to lerp actually produce a kind of exponential decay toward the goal. And for a lerp factor of 0.5, we can see that this pattern is exactly the equation x_t = 0.5^t. So it looks like somehow there is an exponential function governing this relationship, but how did this appear? The trick to uncovering this exponential form is to write our system as a recurrence relation.

The naive `position += (target - position) * factor` breaks at variable frame rates. The frame-rate-independent form solves the underlying ODE exactly:

```js
// halflife in seconds — time for error to halve
position += (target - position) * (1 - Math.exp(-dt * Math.LN2 / halflife));
```

This is the **minimum viable** smoother. It adds latency proportional to the halflife but has zero overshoot. Tools like Mos use this approach: Adjust step, gain, and duration, and see how raw wheel deltas turn into controlled motion. Quantization floor for wheel deltas. Scales distance per tick and how fast the curve ramps. Smoothing time constant (higher means longer tail).

### Tier 2: Critically Damped Spring

A spring is something that tries to maintain a certain length by exerting a force equal to the displacement from that length times a spring constant. A damped spring is one that exerts an additional force proportional to how fast its length is changing in order to counter that motion. When a spring doesn't have much (or any) damping, it oscillates. But if you get the damping just right, the spring will just decay back to its preferred length with no oscillations. This is what's called a critically damped spring.

This is what Firefox actually implemented internally: The MSD-based animation needs velocities to be accurate, otherwise you get weird oscillation effects. So the velocity needs to be computed by the animation and not guessed from the displacement, and the zoom factor needs to be applied.

The critically damped spring is **strictly superior** to exponential decay for scroll because it:
- Carries velocity state — allowing it to respond to new deltas arriving mid-animation without discontinuity
- Can be tuned with intuitive parameters (ω controls "snappiness")
- Has an exact closed-form solution, so it's frame-rate independent

```js
// Critically damped spring update (semi-implicit Euler)
// ω = damping frequency, dt = frame delta
const f = 1 + 2 * dt * ω;
const oo_dt = ω * ω * dt;
const det = 1 / (f + oo_dt);
velocity = (velocity + oo_dt * (target - position)) * det;
position = position + dt * velocity;
```

### Tier 3: Velocity-Accumulating Kinetic Model

iOS-style momentum scrolling uses a velocity accumulator: This observation led me to believe that the momentum scrolling is a sort of exponential decay. It is characterized by the speed of the decay. There are two different ways to express it: half-life or time constant. For the latter, it is very much related to the step response of a first order system. In other words, the deceleration system is just an overdamped spring-mass system. Turns out, everything is still based on physics.

In fact, this is how the deceleration is implemented in Apple's own PastryKit library (and now part of iAd). It reduces the scrolling speed by a factor of 0.95 for each animation tick (16.7 msec, targeting 60 fps). This corresponds to a time constant of 325 msec.

For discrete wheel input, the state-of-the-art combines these: each detent **adds to the current spring velocity** rather than setting a new target position. This means rapid wheel clicks compound naturally (the spring is already moving), while slow isolated clicks produce modest, smooth movements.

## 4. Predictive Delta Extrapolation

The key insight from filtering and interpolation introduce an additional delay in the input processing, we need to make a reasonable tradeoff between the smoothness and the input lag, as we also want to keep the scrolling latency to a minimum.

Practical extrapolation strategies:

1. **Velocity estimation window**: Track the last N wheel events with timestamps. Compute instantaneous velocity as `Σ(delta) / Δt` over a ~80-100ms window. If the user is actively scrolling (events arriving faster than ~120ms apart), bias the spring target forward by `velocity × lookAheadTime`.

2. **Continuation prediction**: If ≥2 wheel events arrive within 150ms, assume the user is in a "scrolling burst" and pre-extend the target by one additional detent's worth of distance. Cancel if no event arrives within the expected interval.

3. **Adaptive halflife**: Shorten the spring's time constant as velocity increases (matching the acceleration curves used by tools like BetterTouchTool): Applies an acceleration curve that makes faster scrolling even faster while maintaining precision for slow scrolls. Formula: output = input × (1 + strength × √(|input|/4)).

## 5. Scheduling: rAF-Aligned vs postMessage Microtask

### rAF-Aligned Batching (Standard Approach)

The `window.requestAnimationFrame()` method tells the browser you wish to perform an animation. It requests the browser to call a user-supplied callback function before the next repaint. The frequency of calls to the callback function will generally match the display refresh rate.

This is the correct default. Your smoothing loop ticks at vsync, DOM writes happen right before paint, and with `requestAnimationFrame`, you update before paint. That means smoother UI, lower perceived latency, and a faster-feeling app.

The inherent **input-to-pixel latency floor** with pure rAF is ~1 frame (worst case: event arrives just after rAF fires; you wait nearly a full frame). When using something like the mouse move event to track the mouse cursor and using requestAnimationFrame for rendering you get a very perceptible input lag of 1 frame between the last available mouse position and rendering.

### postMessage Scheduling (The React Trick)

React discovered this for its scheduler: In React 16.4.0, it was replaced with a combination of `requestAnimationFrame` and `postMessage`.

The reason: If you keep using `setTimeout` to schedule tasks, after a few recursive arrangements, the shortest execution interval will become 4ms, regardless of how much you set the interval. On the other hand, `postMessage` and message handler do not have this limitation, which is why this approach was chosen.

The pattern:
- **rAF** fires → record the frame deadline timestamp
- **Immediately `postMessage`** to self → message handler fires *after* rAF but *before* paint (in practice, within the same frame's task processing)
- Calculate how much time can be spent executing tasks without interfering with rendering in rAF. Execute tasks as much as possible in the message handler.

**However, there's a downside**: The current usage is `window.addEventListener('message', fn)`, so every time a task is scheduled, `window.postMessage` must be used. If there are other listeners on the page, it will be triggered repeatedly. For example, some extensions might print out all received messages to help with debugging, potentially receiving one every 30ms, which could flood the logs. Such side-effect behavior is clearly not ideal and can interfere with other implementations.

### The Tradeoff Matrix

| Strategy | Min Latency | Jank Risk | Complexity | Best For |
|---|---|---|---|---|
| **Pure rAF** | ~1 frame (≤16.6ms @60Hz) | Low | Low | Most scroll smoothing |
| **postMessage after rAF** | Sub-frame (~1-4ms) | Medium | Medium | When you need work *between* frames |
| **MessageChannel** | Sub-frame (~0.1ms) | Medium | Medium | Private channel, avoids window.postMessage pollution |
| **Wheel handler → immediate CSS transform** | ~0ms to compositor | Highest | Low | Compositor-friendly properties only |

For scroll smoothing specifically, **pure rAF is almost always correct**. The postMessage trick helps when you have *work scheduling* concerns (like React's reconciler), not when you're animating a single scroll position. The bottleneck is the rendering pipeline, not the scheduling granularity—your transform/scroll write can only produce a pixel at the next vsync regardless.

### The Real Latency Win: Compositor Thread

The single biggest latency reduction comes from avoiding main-thread scroll entirely. Use `transform: translateY()` on a composited layer with `will-change: transform`. This lets the compositor move the layer independently of the main thread. Combine with the spring physics running in rAF, and you get:

1. Wheel event → main thread accumulates delta
2. rAF → spring physics tick → write `transform` 
3. Compositor composites the layer (potentially with one frame of prediction)

For the absolute lowest latency, the `scroll-behavior: smooth` CSS property or native `scrollTo({ behavior: 'smooth' })` delegates entirely to the browser's compositor-thread scroll animation, which has access to sub-frame timing that JavaScript cannot match. But you lose control over the physics curve.

## Summary: The Recommended Architecture

```
wheel event (passive: false)
  │
  ├─► accumulate delta into velocity state
  ├─► classify: single tick vs. burst (track inter-event timing)
  ├─► if burst: apply acceleration curve to delta
  └─► request rAF (coalescing guard: only once per frame)
          │
          rAF callback
            ├─► read accumulated velocity
            ├─► tick critically-damped spring (exact integration, frame-rate independent)
            ├─► write CSS transform (compositor-friendly)
            └─► if spring still moving: schedule next rAF
```

This gives you: perceptually continuous motion from discrete input, sub-pixel precision via transforms, automatic velocity compounding for rapid scrolling, and the lowest achievable latency within the browser's rendering pipeline constraints.

## 1. Deceleration Curve Models in Practice

The engineering community has converged on a few models through empirical tuning rather than formal psychophysical experimentation:

- **Exponential decay** is the dominant model for momentum scrolling. The observation that momentum scrolling is a sort of exponential decay is well-established — the deceleration system is essentially an overdamped spring-mass system. This was the approach pioneered in early iOS-era implementations (FlickCharm, Zynga Scroller, iScroll, etc.).

- **Cubic Bézier curves** are used as a practical CSS-friendly *approximation* of those physics-based curves. For applying proper transitions to momentum movement, a cubic Bézier curve for slow deceleration ending in full stop (velocity is zero) is a common approach. The look of the Bézier function will be different for each end velocity — on the first picture, deceleration takes content to zero end velocity; on the second, end velocity is bigger than zero; on the third, end velocity equals start velocity, so "the less end velocity will differ from start velocity, the more flat cubic Bézier function will be."

- **Combined curves** (acceleration → coasting → deceleration): Due to a combination of the initial acceleration and the subsequent "coasting," the resulting trajectory closely resembles the cubic ease in/out curve.

However, **none of these sources report A/B testing or user satisfaction ratings.** The curve choices were arrived at through developer intuition, physical analogy, and iterative feel-tuning — not controlled psychophysical experiments.

## 2. Apple's Rubber-Band Scrolling Model

Apple's rubber-band effect uses two distinct mechanics:

- **Momentum deceleration** (within bounds): essentially exponential velocity decay.
- **Rubber-band resistance** (at bounds): The formula Apple uses, as reverse-engineered from iOS, is `b = (1.0 - (1.0 / ((x * c / d) + 1.0))) * d` where `c = 0.55` and `d` equals the screen dimension. This is a rational function that produces diminishing returns — not a spring model per se.

- **Bounce-back** uses a spring animation. When content hits a bound, a spring animation is triggered, and "due to the fact that we use the same concept of velocity in deceleration and spring animation, the junction of these two animations turned out to be smooth."

- Attempts to replicate this with simple functions like `sqrt(x)` failed: By interacting with the Control Centre behaviour, "the first mathematical function that came to my mind was the square root. However… it didn't feel right, and not as natural as Apple's." A logarithmic function (log₁₀) was found to work better.

- There is no information about the rubber band effect in Apple's documentation. This is entirely reverse-engineered.

## 3. Psychophysics of Perceived Smoothness

Formal psychophysics studies motion perception generally, but hasn't been applied to scroll deceleration curves specifically:

- Psychophysics quantitatively investigates the relationship between physical stimuli and the sensations and perceptions they produce, described as "the analysis of perceptual processes by studying the effect on a subject's experience or behaviour of systematically varying the properties of a stimulus along one or more physical dimensions." This methodology *could* be applied to scroll curves, but hasn't been in published literature.

- The perceptual momentum literature examines how humans extrapolate motion: A moving target's continuation or extrapolation continues until momentum is dissipated, and "acceleration (in which momentum increases) or deceleration (in which momentum decreases) leads to a larger or smaller momentum-like effect." This is about *representational* momentum in perception, not UI scroll evaluation.

- The 60fps threshold is a widely cited benchmark: An FPS value of 60 corresponds to a smooth visual experience for the human eye, meaning a frame should take no longer than 16ms to load.

## The Core Problem: Sticky + translate3d + Scroll = Layout Storm

You're describing a pathological rendering scenario. Here's why it's so expensive:

1. **Sticky position recalculation is per-scroll-frame.** Scrolling elements containing fixed or sticky content can cause performance and accessibility issues. As a user scrolls, the browser must repaint the sticky or fixed content in a new location. Depending on the content needing to be repainted, the browser performance, and the device's processing speed, the browser may fail to hit 60fps.

2. **`transform: translate3d` on siblings creates a containing block conflict.** There is a known bug with CSS that stops `position: fixed` from working when the element or parent has been transformed. This applies analogously to sticky elements—a `transform` on an ancestor establishes a new containing block, which can alter (or break) the sticky constraint rectangle, forcing the browser to recompute the sticky offset against a different coordinate space each frame.

3. **Forced synchronous layout (reflow) is the killer.** A forced reflow occurs when JavaScript queries geometric properties (such as `offsetWidth`) after styles have been invalidated by a change to the DOM state. This forces the browser to immediately do a layout, which interrupts script execution and results in poor performance.

---

## SOTA Techniques to Prevent Synchronous Layout from Sticky Recalculation

### 1. Batch Reads Before Writes (The Fundamental Rule)

For loops that force layout & change the DOM are the worst—avoid them. Batch your writes & reads to the DOM (via FastDOM or a virtual DOM implementation). Read your metrics at the beginning of the frame (very very start of rAF, scroll handler, etc), when the numbers are still identical to the last computed layout.

**Pattern:**
```js
// ✅ Read phase (no reflow if layout is clean from last frame)
const offsets = stickyHeaders.map(h => h.parentElement.getBoundingClientRect().top);

// ✅ Write phase (batch all mutations)
requestAnimationFrame(() => {
  stickyHeaders.forEach((h, i) => {
    h.style.transform = `translateY(${computeStick(offsets[i])}px)`;
  });
});
```

Reading positioning and styling information can be quite fast if performed on a fully cached, styled and positioned Render Tree. Writing positioning information will mark nodes as dirty, but won't force the browser to perform reflow immediately.

### 2. CSS `contain: layout style` on Scroll Sections

The `contain` property allows an author to specify exactly what containment types they want applied to individual containers on the page. This allows the browser to recalculate layout, style, paint, size, or any combination of them for a limited part of the DOM.

```css
.scroll-section {
  contain: layout style paint;
}
```

**Measured frame budget impact:** Performance test results (avg. frame time) — CSS Sticky: 1.2ms, Intersection Observer: 1.8ms, Scroll Event Handler: 4.3ms, Transform Animation: 6.1ms.

Applying `contain: layout style` limits the **scope of reflow** to the contained subtree. In the pathological N-sticky-headers case, without containment, a single sticky recalculation can invalidate the entire document layout. With containment, each section's sticky computation is isolated. In practice this means:

- **Without `contain`:** Sticky recomputation ≈ O(N × M) where M is total DOM size
- **With `contain: layout style`:** Sticky recomputation ≈ O(N × Mₛ) where Mₛ is the section subtree size

The absolute numbers from benchmarks suggest **~1.2ms per frame for pure CSS sticky** with containment, well within the ~10ms JS budget of a 16.6ms frame. But the key win is that `contain` prevents the absolutely-positioned `translate3d` children from invalidating the sticky ancestor's layout scope.

> ⚠️ **Critical caveat:** `contain: layout` on the sticky element's *parent* can **break** sticky behavior entirely, because containment changes the containing block semantics. Apply `contain` to **sibling containers**, not to the sticky element's direct scrolling ancestor.

### 3. `will-change: transform` for Compositor Promotion

One solution is to add `will-change: transform` to the positioned elements to render the element in its own layer, improving repaint speed and therefore improving performance and accessibility.

```css
.sticky-header {
  position: sticky;
  top: 0;
  will-change: transform;
}

.abs-child {
  position: absolute;
  transform: translate3d(var(--x), var(--y), 0);
  will-change: transform;
}
```

This promotes both sticky headers and absolute children to **compositor layers**, meaning their visual updates can bypass the main-thread layout pipeline entirely during scroll. The sticky offset is still computed on main thread, but the actual pixel movement is a GPU-side compositing operation.

### 4. `content-visibility: auto` for Off-Screen Sections

The `content-visibility` property allows authors to apply a strong set of containments on a set of containers and specify that the browser should not lay out and render those containers until needed. A second property, `contain-intrinsic-size`, is also available, which allows you to provide a placeholder size for containers while they are under the effects of containment. This means that the containers will take up space even if their contents have not yet been rendered.

```css
.scroll-section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px; /* estimated height */
}
```

This skips layout entirely for off-screen sections—if you have 50 sections with sticky headers, only the ~3-5 visible ones participate in layout at all.

## Can IntersectionObserver Replace Per-Frame Sticky Recalculation?

**Partially yes, but with critical caveats:**

### What IO Can Replace
IO excels at detecting **state transitions** (stuck vs. unstuck). To take it a step further, we wanted to optimize for performance. Having an event listener running constantly on the page was not performative at all, so I looked into other options. That's where Intersection Observer API came in handy. The Intersection Observer API provides a way to asynchronously observe changes in the intersection of a target element with the viewport.

The **sentinel pattern** is the SOTA approach:
```js
// Place zero-height sentinels above each sticky header
const sentinel = document.createElement('div');
section.prepend(sentinel);

const observer = new IntersectionObserver(
  ([entry]) => {
    entry.target.nextElementSibling.classList.toggle('is-stuck', !entry.isIntersecting);
  },
  { threshold: [0], rootMargin: '-1px 0px 0px 0px' }
);
observer.observe(sentinel);
```

### What IO Cannot Replace

You really want to use `position: sticky` for this. Scroll events and Intersection Observers are async, and will cause elements to jump if you try to toggle their position in JavaScript (depending on device).

The fundamental limitation is:

| Concern | IO | CSS `position: sticky` |
|---|---|---|
| **Timing** | Asynchronous (fires between frames) | Synchronous with scroll compositor |
| **Pixel-precision positioning** | ❌ Cannot set sub-frame offsets | ✅ Browser handles natively |
| **State detection** (stuck/unstuck) | ✅ Excellent with thresholds | ❌ No native CSS event |
| **Frame budget** | ~0.1-0.3ms per callback | ~1.2ms per frame (with containment) |
| **Works with translate3d siblings** | ✅ (doesn't care about containing block) | ⚠️ Can break if transform is on ancestor |

**Threshold arrays** (`threshold: [0, 0.1, 0.2, ..., 1.0]`) give you quantized intersection ratios, but even at 100 thresholds you get ~1% granularity. This is sufficient for:
- Toggling visual states (shadow, background)
- Activating/deactivating headers in a TOC
- Lazy-loading content as sections approach

But it is **not** sufficient for smooth per-pixel sticky positioning. If you're using JS to *calculate* sticky offsets (e.g., stacking multiple sticky headers), you still need the native CSS sticky mechanism or a `requestAnimationFrame` loop.

### The Hybrid Architecture (Recommended SOTA)

```css
.sticky-header {
  position: sticky;
  top: 0;
  will-change: transform;
  contain: style;         /* scope style recalc */
}

.scroll-section {
  contain: layout style paint;
  content-visibility: auto;
  contain-intrinsic-size: 0 600px;
}

.abs-positioned-child {
  position: absolute;
  transform: translate3d(var(--x), var(--y), 0);
  will-change: transform;  /* own compositor layer */
}
```

```js
// IO for state transitions only (class toggling, not positioning)
const io = new IntersectionObserver(
  (entries) => {
    for (const entry of entries) {
      entry.target.dataset.stuck = !entry.isIntersecting;
    }
  },
  { threshold: [0, 1], rootMargin: '-1px 0px 0px 0px' }
);

sentinels.forEach(s => io.observe(s));

// NO per-frame scroll handler needed for position
// CSS sticky + compositor layers handle it
```

### Summary Table: Frame Budget Impact

| Technique | Typical Frame Cost | Prevents Forced Reflow? |
|---|---|---|
| Pure CSS `sticky` (no containment) | ~2-4ms | N/A (no JS) |
| CSS `sticky` + `contain: layout style` | **~1.2ms** | N/A (no JS) |
| IO sentinel (state toggle only) | **~0.1-0.3ms** per callback | ✅ Yes |
| `scroll` event + `getBoundingClientRect` | ~4.3ms+ | ❌ No — forces reflow |
| JS `transform` animation in rAF | ~6.1ms | ❌ If reads precede writes |
| `content-visibility: auto` on sections | **Eliminates** off-screen cost | ✅ Yes |

Avoid forced synchronous layouts and layout thrashing; read style values then make style changes. Layout is normally scoped to the whole document. The number of DOM elements will affect performance; you should avoid triggering layout wherever possible.

**Bottom line:** Let CSS `position: sticky` handle positioning (it runs in the compositor where possible). Use `contain: layout style paint` on section wrappers to scope reflow. Use `content-visibility: auto` to skip off-screen sections entirely. Use `IntersectionObserver` only for discrete state transitions (stuck/unstuck class toggling). Never read geometric properties inside a scroll handler after DOM mutations—that's where the synchronous layout thrashing hides.

## The Core Conflict

Passive listeners let the scroll thread keep working without getting blocked. This lets the compositor thread update scroll positions right away, even though the event handler still runs later on the main thread. The moment you need `preventDefault()` to implement custom momentum physics, you lose this guarantee. Even if your scroll handler is doing something light, the browser doesn't want to take the risk. Without a passive listener, it holds back a frame just in case you cancel it. With one, it skips that wait.

The deeper architectural reason: the compositor thread marks a region of the page that has event handlers attached as "Non-Fast Scrollable Region." By having this information, the compositor thread can make sure to send input event to the main thread if the event occurs in that region. If input event comes from outside of this region, then the compositor thread carries on compositing new frame without waiting for the main thread.

It is not feasible to prevent all these stalls: style recalculation, synchronous network requests, long painting times, garbage collection, all these things have content-dependent costs. The compositor thread architecture allows us to snapshot a version of the page and allow the user to scroll and see animations directly on the snapshot, presenting the illusion that the page is running smoothly.

So **any architecture that needs to intercept native scroll deltas on the main thread before they're applied is fundamentally at odds with compositor scrolling.** Here are the viable patterns, from best to worst:

---

## Pattern 1: CSS-Only Scroll-Driven Animations (Best — Full Compositor)

If your "custom physics" is really about visual responses to scroll position (parallax, transformations, reveal effects), the modern answer is to never touch JS at all:

What we have now is no JS, no dependencies, no libraries — just pure CSS. And if that's not exciting enough, these animations run off the main thread, delivering smooth, high-performance, GPU-accelerated experiences.

Scroll-driven animations work off the main thread when animating properties that can be animated on the compositor like transforms and opacity, whether you're using the new API directly in CSS or using the JavaScript hooks. Real-world results are dramatic: Tokopedia "managed to reduce up to 80% of our lines of code compared to using conventional JS scroll events and observed that the average CPU usage reduced from 50% to 2% while scrolling."

**Limitation:** This doesn't help when you need to change *how the scroll offset itself is computed* — it only lets you respond to an existing scroll position with visual effects.

---

## Pattern 2: `touch-action: none` + Pointer Events + `transform: translate()` (Your Best Bet for True Custom Physics)

This is the most architecturally sound way to **fully replace native scroll behavior** without blocking the compositor for native scroll, because you *eliminate* native scrolling rather than fighting it:

**How it works:**

```css
.scroll-container {
  touch-action: none;       /* Kill native gestures at the compositor level */
  overflow: clip;            /* No scroll container created at all */
  position: relative;
  will-change: transform;
}
```

```js
// Pointer events give you raw input without fighting the compositor
container.addEventListener('pointerdown', onDown);
container.addEventListener('pointermove', onMove);  // passive is fine
container.addEventListener('pointerup',   onUp);

// For wheel input on desktop
container.addEventListener('wheel', onWheel, { passive: true });

// Your physics simulation runs entirely in rAF
function tick(ts) {
  physics.step(ts);
  content.style.transform = `translate3d(0, ${-physics.offset}px, 0)`;
  requestAnimationFrame(tick);
}
```

**Why this works:** Because it influences whether the browser will perform native gesture handling, touch-action has a direct impact on event timing and the need for JavaScript to call preventDefault() on touch/pointer events. When the browser knows a gesture is allowed, it can handle it natively without waiting to see if script will cancel it, which reduces input latency.

Critically, `touch-action: none` is a **declarative signal** processed at the compositor level before event dispatch. Setting `touch-action` to `none` will disable all browser handling of these events, leaving them up to you to implement (via JavaScript). If you only want to take over one interaction, tell the browser to handle the rest. The compositor never has to wait on main-thread `preventDefault()` — it already knows not to scroll.

**Why `overflow: clip` over `overflow: hidden`:** overflow: clip works exactly the same as overflow: hidden, except for the fact that overflow: clip does not create a scroll container. This matters because when you set `overflow: hidden`, it also creates a scroll container, which means you might get ghost scroll events, interfere with scroll-driven animation lookups, and leave the compositor in ambiguous state about who "owns" scrolling.

**The full architecture looks like this:**

```
┌──────────────────────────────────────────────────────────┐
│  COMPOSITOR THREAD                                        │
│                                                          │
│  touch-action: none → no native gesture handling          │
│  overflow: clip     → no scroll container exists          │
│  Compositor has NOTHING to scroll — it's not blocked,     │
│  it's simply not involved in scroll at all.               │
│                                                          │
│  transform: translate3d() → composited layer move         │
│  (GPU-accelerated, no layout/paint)                       │
└──────────────┬───────────────────────────────────────────┘
               │ pointer events dispatched normally
┌──────────────▼───────────────────────────────────────────┐
│  MAIN THREAD                                              │
│                                                          │
│  pointerdown/move/up → feed velocity into physics engine  │
│  wheel events (passive) → feed delta into physics engine  │
│                                                          │
│  rAF loop:                                                │
│    physics.step(dt)                                       │
│    content.style.transform = translate3d(0, -offset, 0)   │
│    — OR —                                                 │
│    content.style.transform via WAAPI for even less        │
│    main-thread paint overhead                             │
└──────────────────────────────────────────────────────────┘
```

**Trade-offs:**
- You **must** implement your own momentum/friction/spring physics (but that's what you wanted)
- You lose native scrollbar (must synthesize if needed)
- You lose native overscroll/rubber-banding (must implement)
- `position: sticky`, CSS `scroll-snap`, Scroll-Driven Animations, etc. won't work — those expect a real scroll container
- `touch-action: none` may inhibit operating a browser's zooming capabilities. This will prevent people experiencing low vision conditions from being able to read and understand page content. Consider `touch-action: pinch-zoom` if you only need to intercept panning.

## Pattern 3: Your Proposed `overflow: hidden` + `scrollTo()` in rAF

This is viable but has subtle problems:

```css
.viewport {
  overflow: hidden;    /* Suppresses visible scrollbar and user-initiated scroll */
  height: 100vh;
}
.content {
  /* Tall enough to have a scrollable range */
  height: 5000px;
}
```

```js
// Feed input into physics engine, then:
function tick() {
  physics.step();
  viewport.scrollTo(0, physics.offset);  // programmatic scroll
  requestAnimationFrame(tick);
}
```

**Problem 1:** `overflow: hidden` creates a scroll container. The content "must still be scrollable programmatically [...] and the box is therefore still a scroll container." This means the compositor *knows* this is scrollable, and on some platforms (especially mobile), you may get interference — the compositor might try to handle touch gestures on this scroll container before your pointer events fire.

**Problem 2:** `scrollTo()` triggers a full scroll event cycle — it updates `scrollTop`, fires `scroll` events, and may cause layout recalculation if anything depends on scroll position. On each rAF you're doing a "real" scroll operation on the main thread, which is heavier than setting a `transform`.

**Problem 3:** Because a real scroll container exists, you still need non-passive touch listeners or `touch-action: none` to prevent the native scroll from fighting your programmatic `scrollTo()`. You haven't avoided the core conflict — you've just moved it.

**When it IS useful:** If you need things that depend on a real scroll container — like `position: sticky`, `IntersectionObserver` with a scroll root, or native `scrollbar-*` styling — then this pattern gives you a real scroll position to work with. The trade-off is that you're on the main thread for scroll updates.

## Pattern 4: Passive Listener + Post-hoc Correction (Compromise)

If you want to *mostly* keep compositor scrolling but apply corrections:

```js
element.addEventListener('scroll', () => {
  // Read native scroll position (compositor already moved the content)
  const native = element.scrollTop;
  
  // Apply your custom physics offset as a CSS transform correction
  const correction = customPhysics.computeCorrection(native);
  content.style.transform = `translateY(${correction}px)`;
}, { passive: true });
```

Most browsers now support some sort of asynchronous scrolling in order to provide a consistent 60 frames per second experience to the user. In the asynchronous scrolling model, the visual scroll position is updated in the compositor thread and is visible to the user before the scroll event is updated in the DOM and fired on the main thread.

This means there's always a **1-frame visual desync** — the compositor applies the native offset instantly, then your main-thread correction arrives one frame later. The event handler will be delayed relative to the user-visible scroll, and so the div will not stay visually fixed as intended. Instead, it will move with the user's scrolling, and then "snap" back into position. This constant moving and snapping will result in a jittery visual effect. For subtle corrections (e.g., variable-speed scrolling where the correction is small relative to the delta), this shimmer is barely perceptible. For full custom momentum, it's unusable.

## Pattern 5: The Houdini Scroll Customization Dream (Not Yet Real)

The fundamental problem here is that scrolling on the web is a big "magical" black-box. In other UI frameworks scrolling is usually implemented as a library (with extensibility points) on top of primitives. The goal of the Houdini Scroll Customization proposal is to break open this black box to make scrolling on the web extensible.

Scroll Customization introduces a new API for content to dictate how a scroll delta is applied and consumed. As of this writing, Mozilla does not plan to support this proposal. This API would let you run custom physics *on the compositor thread* via a worklet. It's essentially dead in standards at this point.

## Recommendation

For true custom momentum physics, **Pattern 2 is the correct architecture:**

| Concern | Solution |
|---|---|
| Prevent native scroll | `touch-action: none` (declarative, compositor-aware) |
| No ghost scroll container | `overflow: clip` (not `hidden`) |
| Visual movement | `transform: translate3d()` (compositor-composited) |
| Input capture | Pointer Events (not touch events) |
| Wheel input | Passive `wheel` listener, feed delta to physics |
| Physics simulation | rAF loop with your spring/friction/momentum model |
| GPU promotion | `will-change: transform` on content layer |

This gives you the best of both worlds: the compositor is never blocked (because there's nothing for it to scroll), and your visual updates go through the fastest composited path (transforms). You're trading native scroll amenities (sticky, snap, scrollbars, accessibility) for full control — which is exactly the trade-off custom scroll physics demands.

**Frame time consistency (pacing) is far more perceptually important than average FPS for scroll smoothness**, as you noted. A stable 50 FPS often feels better than a jittery 60 FPS because human visual perception is highly sensitive to timing irregularities, micro-stutters, and latency spikes during continuous motion like scrolling.

### Key Metrics Beyond Average FPS
These metrics better predict perceived scroll smoothness:

- **Frame time variance, standard deviation, or jitter**: Measures how much individual frame durations deviate from the ideal (e.g., ~16.67 ms at 60 Hz or ~8.33 ms at 120 Hz). Low variance produces the "buttery" feel; high variance creates visible hitching even at high average FPS.

- **Percentile frame times (especially p95/p99) and maximum frame interval**: These capture outlier "bad frames" or spikes that the brain notices most. A single long frame (e.g., 30–50+ ms) disrupts the flow more than a slight average drop. Studies identify the *maximal frame interval* as one of the strongest predictors of perceived smoothness.

- **Jank rate / missed VSyncs / "Janks" count**: The number or percentage of frames that miss the display's refresh deadline (VSync). Android defines "Janks" based on how much a frame exceeds the vsync period (e.g., >1x, >2x the ideal interval), providing a direct count of perceptible stutters. Tools like Perfetto/Systrace quantify this during scrolls.

- **Input-to-display (motion-to-photon) latency percentiles**: Time from scroll input (wheel, touch, mouse) to the corresponding visual update on screen. Median latency matters for responsiveness, but 95th/99th percentile tail latency drives frustration during sustained scrolling. High values here make scrolling feel "laggy" or detached even if frame rate looks okay.

- **Jank budget utilization**: How much of the per-frame time budget (refresh interval minus overhead) is consumed by pipeline stages—especially main-thread work (JavaScript, style/layout, paint) vs. compositor/GPU work. High utilization (>70–80% sustained) risks frame drops under variable load. In browsers, this often shows up as main-thread blocking during scrolls.

Additional supporting signals include frame time histograms (ideal: tight clustering around the target interval) and compositor thread stability (whether scrolling stays fully off the main thread).

### Published Work on Perceptual Scroll Quality
There isn't a single universal, standardized "perceptual scroll quality score" (analogous to video SSIM or gaming QoE models) that fully integrates all factors, but several studies and engineering frameworks provide strong foundations:

- **Smartphone/UI smoothness perception studies** (e.g., Tan et al., 2021): Experimental work on frame-loss (delayed or dropped frames leading to longer intervals) in smartphone operations found that **single frame-loss number (SFLN)**, **frame-loss time (FLT, i.e., duration of bad frames)**, and their interactions significantly affect perceived smoothness. Maximal frame interval and the number of intervals exceeding thresholds were among the most critical indexes—outperforming simple average frame rate. This applies well to scrolling as a continuous gesture-driven interaction.

- **Variable frame timing (VFT) research** (NVIDIA, 2024): In motion-intensive scenarios (relevant to scrolling), larger frame time variations (~12 ms) noticeably degrade perceived smoothness, while smaller variations (~4 ms) have diminished impact. This directly informs VRR behavior.

- **Chromium/Chrome jank analysis**: Engineering documentation categorizes scrolling jank symptoms (e.g., checkerboarding from rasterization lag during fast scrolls, latency from touch/wheel handlers, framerate drops from main-thread work) and ties them to measurable pipeline delays via tracing (frame times > VSync interval, input-to-frame gaps).

- **Android performance tooling**: Systrace/Perfetto includes dedicated smoothness/jank analysis that correlates frame scheduling, VSync misses, and user-perceived jitter thresholds. Different people have varying sensitivity, but consistent small jitter is generally better tolerated than infrequent large spikes.

Related HCI and gaming QoE work (e.g., on frametime spikes during specific actions or latency jitter) reinforces that context (e.g., during active scrolling/targeting) modulates sensitivity, and tail latencies hurt more than averages.

Browser vendors and OS teams rely on internal telemetry, A/B testing with real users, and tools like Chrome DevTools (Performance panel for frame timing, Layers panel, Rendering tab) rather than a single published scalar score.

### Content Complexity, Compositing Layers, Display Refresh Mismatches, and Webviews
**Content complexity interacts strongly with these metrics**:

- **High DOM node count**: Increases style recalculation, layout, and paint costs. If scroll handlers or dynamic content trigger these on the main thread, it directly inflates frame times and jank budget usage, leading to missed frames or higher variance. Virtualized lists (e.g., React Window, virtual scrolling) or reducing unnecessary nodes help.

- **Compositing layer count**: Browsers promote elements (e.g., via `transform`, `will-change: transform`, or overflow scrolling containers) to GPU layers for fast scrolling without main-thread repaints. Too few layers → paint work on scroll → jank. Too many layers → higher memory/bandwidth use during compositing, potential overdraw, and GPU pressure. Optimal management (via DevTools Layers panel) is key for smooth scrolling. Over-promotion can hurt more than it helps on lower-end devices.

**Variable Refresh Rate (VRR) and mismatches**:
- VRR (G-Sync/FreeSync/Adaptive-Sync, ProMotion, etc.) helps by dynamically matching display refresh to produced frames, reducing tearing and some stutter from rate mismatches. However, it does *not* eliminate the perceptual penalty of high frame time variance or large spikes—per the NVIDIA VFT study. Inconsistent production still feels bad; VRR is best paired with good pacing.
- In web browsers and webviews, support varies (some Edge experiments, proposals for better web VRR integration). Fixed-rate web content on a VRR display can still exhibit issues if the compositor doesn't adapt perfectly. Fast scrolling benefits more from consistent high-rate production than pure VRR magic.

**Webview-composited environments** (e.g., Android WebView, iOS WKWebView, Electron, React Native WebView):
- Add an extra compositing layer between web content and native UI. This can introduce synchronization overhead or extra latency. The web rendering pipeline (with its DOM/layer costs) still dominates internal smoothness, but the outer compositor must handle integration without adding jank. Optimizations focus on the same web metrics plus minimizing cross-boundary work (e.g., avoid heavy native-web bridging during scrolls).

In practice, no off-the-shelf formula perfectly scores the full interaction (DOM count × layers × VRR mismatch), but you can approximate one via regression on user study data or telemetry: weight frame time stats heavily, add penalties for high main-thread utilization correlated with node/layer counts (measurable in traces), and adjust for display refresh characteristics. Tools like Chrome Tracing, Perfetto, or custom instrumentation during scrolls provide the raw data.

For implementation, profile real content on target devices/displays (including VRR modes), target consistent sub-16 ms (or sub-8 ms at 120 Hz) frame times with low variance, keep scroll handlers passive and compositor-only where possible, and validate with user perception tests rather than FPS counters alone. This is an active area in browser and mobile performance engineering.
